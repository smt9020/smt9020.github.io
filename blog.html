Service Test Data Driving This article was really really helpful, especially because a lot of what I was doing when I first starting working with UFT was surrounding API and Service testing. He misses something at the end of step 3. He says that you have to go into the Test Configurations > Data tab in order to mark your test as dynamic and reference the datasheet. Which is all correct. But THEN you have to click the map button on the right of that which he doesn't show, and map all your parameters again. If you don't do that, then it will always use the values from the datasheet you updated to UFT, and it won't run based on your updates to Test Resources. But once you enter in those mapping values you might see something like this.... Index was out of range. Must be non-negative and less than the size of the collection ..... THAT means that you need to delete the mappings that you just put in. Good news is, you won't have the problem of your test resources not updating anymore. Redirected Output with Log Rotation

I'm trying to output my exe to a file on the command line, so I do something like...

myExecutable.exe > myLog.log

And that totally works. And then I want to search through the log file for any error or disconnect messages from my executable, something like...

@echo off

for %%i in (.\myLog.log) do (

FINDSTR "error" %%i >> myfinds.txt

)

This also works. Now I have all of my error statements in one myFinds.txt file (The logging has timestamps associated with each line, so if I can pull out the errors I can look at it in one place and go to the other log files as needed to see what was around the failures). The problem is that the myLog.log file gets really large; too large to maintain. So after I do my find I want to empty the file out or rename it to something like myLog.log.1272016. When I do that, I get this error message:

This process cannot access the file because it is being used by another process

The executable is still constantly pumping output data into myLog.logso I can't grab ahold of it to change it in anyway. One would think, of course there is an easy fix on te internet! But I haven't been able to find one, so here's what I got so far:

  • tee command for windows. Gotta love StackOverflow

    • I could send my output to multiple files this way? But I'm not sure if I would then be able to edit a subset while still running my exe. Wait also could I not just do this in powershell?

  • Success! Create a C# solution that will set a rollover limit

    • Gather more info on this...

  • Success?logrotate (on linux but I have Windows so logrotatewin)

    • I found this online, and it looks encouraging. but in order to use the windows file there is setup.exe I have to run? which seems a little shady

  • Stop the .exe rename my log file and start the .exe

    • The exe I'm running will keep running until I cancel it. So if I stop it, quickly rename my file, and restart it, it doesn't change what the executable is actually doing. Bad Agile

Ways Agile can be flawed

In this article the author points out 3 agile principles that can be interpretted in a way that can hinder your project's progress.

I have heard #1, delivery over quality, from people over and over again and have seen it myself. Getting things to a deliverable state as soon as possible can lead to a line of thinking that ignores the future. You are expected to develop exactly to the story description and nothing more. Then when it's time to do another story you have to go back and rework what was done in story #1 to figure out how you will add in story #2.

Another point of this article: "agile promotes the practice of ignoring defects". When a defect is found, it is thrown into an endless queue of defects. So then you end up adding on iterations to deal with all the defects, or simply "deferring" them in order to fit in new features. Whether or not those deferred defects ever surface again is on a team to team basis I would imagine.

His third point, collaboration over management, I have not experienced; of teams I have been on most of the Scrum Masters have had a pretty firm grip on the direction of the team and the project. I have seen a lack of collaboration amongst the different team roles. I have primarily sat as a QA rep on agile teams, and have seen struggles with collaborating with the whole of the QA department (as to ensure consistency and collaboration amongst projects). I have also seen teams struggle with the idea of being "told what to test". QA should be deciding to what degree each feature is tested based on their knowledge of the project. Occasionally team members will "suggest" that certain things don't need to be tested together, and in the isolation of an agile team it can be convincing. You will be pressured to agree, and not a team player if you test outside the promoted confines.
