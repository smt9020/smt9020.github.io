Selenium - what makes a good framework?

I'm sure there is a lot more to it then this, but over time I hope to compile a list of things you need for a good Selenium framework. "Good" is going to mean something different to everyone, and is going to depend a lot on who your audience is. In many projects, your audience is the developers who want to be able to test basic functionality of their web app over and over without having to do it manually. It's also common to see selenium wrapped in continuous integration solutions like Jenkins or TeamCity. My audience in this case is the QA group. How can we create a maintainable group of tests that can be run and maintained during regression testing by Test Engineers who do not have experience with coding? There are several building blocks for creating a Selenium framework that BOTH soltutions may find useful.

  • Test Input

    • A way of structuring the parameters of your test so that you can run the same test over and over with different boundary cases. If your audiences is mostly tech-saavy developers, you can do this with command line arguments. If they are not, you can do this with a custom runner, or supplying a spreadsheet that your test can parse. Alternatively you can create a custom connecttion of your Selenium framework to test management solutions such as HP QC, and use the parameter support that comes with that.

  • Object Repository

    • Organizing your page objects so that you can reference them with a friendly-name, and also only have to make a change in one place when a UI change will alter your test or break it. This can be as simple as a class of constants with xpaths to identify different buttons on a screen. It could also take the form of a spreadsheet. Load your spreadsheet into an ObjectRepository object so that you can do things like ObjectRepo("button-name").Click(1). What about sharing my object repositories between pages? How do we know which ObjRepos to load in? ...still investigating...

  • Assertions and Reporting

    • Using NUnit or JUnit for assertions is a pretty easy win. And then you can create your own logging, or get really fancy and use something like log4net.

  • Libraries

    • Broken down by page = Useful more when your audience is development

    Abstract with methods such as "click element" = Useful more when your audience is QA

  • Version Control

    • Obviously you need to version control your Selenium code. But what about controlling which versions of the test work on which versions of the product you are testing? You need to have a way of versioning your object repository and your task flow based on which version of the product that test will work on.

  • Batch Runner

    • Apache ANT for batch execution on many manual testers machines.

  • Screenshots during runs

    • Do we want to take screen shots during our runs of the test? Or at particular key moments? Maybe whenever there is fail? how would this work? Could this be leveraged for language testing?

  • List of Best Practices for Developers

    • There should be a list of best practices for automation developers. See this article Selenium Best Practices. There should also be one for application developers, that outlines things they can do to make automation of the webpages easier. For example, making sure all their elements are associated with unique IDs

  • Every Test should clean itself up

  • Can you use Selenium with an Emulator?

    • It only does web pages, so really I think this would only mean "can you have a small screen" to which the answer is yes?

Using Assemblies in Vbscript and C# (UFT)

These directions were meant for someone using UFT and wanting to also use their own .dlls as part of the code (which can really come in handy). But it is probably also useful for anyone using vbscript that needs to incorporate C# code for some reason. Here are the different wats I found to do this:

VBSCRIPT:

strFile = "C:\\Users\\me\\ShannonPractice.dll"
Set oCustom = DotNetFactory.CreateInstance("MyNamespace.PublicClass", strFile)
MsgBox oCustom.myMethod 'this will give you whatever the method would return

C# (UFT Specific Version):

  1. In the CustomCode test, click View> Solution Explorer
  2. Right click on References and select Add Reference…
  3. In the Add Reference dialog, click on the .Net Assembly Browser tab and navigate to the location where you have your DLL
  4. Select each DLL and click Select
  5. Click Ok
  6. In the TestUserCode.cs, add the appropriate using namespace statements.
  7. Now in TestUserCode.cs you can type something like (new PublicClass().myMethod()) and actually use the methods from the DLL

C# (Load during run time):

using System.Reflection;
…
var myDLL = Assembly.LoadFile(@"C:\ShannonPractice.dll");
var theType = myDLL.GetType("ShannonPractice.PublicClass");
var c = Activator.CreateInstance(theType);
var method = theType.GetMethod("myString");
method.Invoke(c, null);

Redirected Output with Log Rotation

I'm trying to output my exe to a file on the command line, so I do something like...

myExecutable.exe > myLog.log

And that totally works. And then I want to search through the log file for any error or disconnect messages from my executable, something like...

@echo off
for %%i in (.\myLog.log) do (
FINDSTR "error" %%i >> myfinds.txt
)

This also works. Now I have all of my error statements in one myFinds.txt file (The logging has timestamps associated with each line, so if I can pull out the errors I can look at it in one place and go to the other log files as needed to see what was around the failures). Different POV on Agile

Ways Agile can be flawed

In this article the author points out 3 agile principles that can be interpretted in a way that can hinder your project's progress.

I have heard #1, delivery over quality, from people over and over again and have seen it myself. Getting things to a deliverable state as soon as possible can lead to a line of thinking that ignores the future. You are expected to develop exactly to the story description and nothing more. Then when it's time to do another story you have to go back and rework what was done in story #1 to figure out how you will add in story #2.

Another point of this article: "agile promotes the practice of ignoring defects". When a defect is found, it is thrown into an endless queue of defects. So then you end up adding on iterations to deal with all the defects, or simply "deferring" them in order to fit in new features. Whether or not those deferred defects ever surface again is on a team to team basis I would imagine.

His third point, collaboration over management, I have not experienced; of teams I have been on most of the Scrum Masters have had a pretty firm grip on the direction of the team and the project. I have seen a lack of collaboration amongst the different team roles. I have primarily sat as a QA rep on agile teams, and have seen struggles with collaborating with the whole of the QA department (as to ensure consistency and collaboration amongst projects). I have also seen teams struggle with the idea of being "told what to test". QA should be deciding to what degree each feature is tested based on their knowledge of the project. Occasionally team members will "suggest" that certain things don't need to be tested together, and in the isolation of an agile team it can be convincing. You will be pressured to agree, and not a team player if you test outside the promoted confines.
